* Change Destiny Feature 仕様

この文書は、抽選された番号にあとから介入するイベント機能について、
=2026-03-29= 時点で先に固定する仕様をまとめたものである。

今回の段階では、数字操作 API の細かい入出力やバランス調整までは確定しない。
先に「イベントゲージ」「イベント発火モーダル」「抽選番号管理の新データ構造」を固定することを目的とする。

補足:

- =2026-03-30= 時点で、共有イベントゲージ、ゲージ満タン時の自動イベント発火、サーバー側のイベント選択 API までは着手済み
- まだ未実装なのは、フロントエンド側のイベントカード UI と番号遷移アニメーション本体である

* 1. 目的

- 従来の受動的なビンゴに、プレイヤーが関与した結果として発生するイベントを導入する
- イベントによって、いったん抽選された番号をあとから変化させる
- ただし、その後の進行は既存の「マスを開ける or スキップする」流れに合流させる

この機能の狙いは、
=プレイヤーの開封行動がゲーム全体の運命改変につながる=
という手触りを作ることである。

* 2. 今回固定するコア仕様

今回、以下を確定仕様とする。

1. イベントは全プレイヤーが共有している抽選番号に対して作用する
2. イベントゲージはプレイヤー個別ではなく、セッション全体で共有する
3. 各プレイヤーがマスを開けるたびにイベントゲージが増加する
4. イベントゲージが満タンになった場合、そのラウンド終了後ではなく「次ラウンド開始直後」にイベントを自動発火させる
5. イベント発火時にプレイヤーの意思決定は挟まない
6. クライアントはイベントモーダルを表示し、元の番号が変化していく演出を描画する
7. 実際にマス開封判定へ使うのは、イベント後に確定した番号だけである

* 3. ラウンド進行仕様

** 3.1 現在の流れ

現状の基本進行は次のとおりである。

1. ホストが次ラウンド開始
2. サーバーが番号を 1 つ抽選
3. プレイヤーがカードを開ける or スキップする
4. 全員完了後、次ラウンドへ進む

** 3.2 変更後の流れ

変更後は次の流れとする。

1. ホストが次ラウンド開始
2. サーバーは抽選候補から通常の抽選番号を 1 つ決定する
3. もしイベントゲージが満タンでなければ、その番号をそのまま確定番号とする
4. もしイベントゲージが満タンなら、サーバー側で数字操作イベントを自動発火する
5. サーバーはイベント処理後の確定番号を決定する
6. クライアントはモーダルで =元の番号 -> 変化後の番号= を表示する
7. クライアントは確定番号をもとに、マスを開けるかスキップする
8. プレイヤーがマスを開けたら、そのたびにイベントゲージを加算する
9. 全員完了後は既存の流れに合流する

補足:

- イベントゲージが満タンになった瞬間にその場で割り込ませるのではなく、次ラウンド開始時に自動発火する
- そのため、あるラウンドの開封結果が「次ラウンドの運命」に影響を与える構造になる

* 4. イベントゲージ仕様

** 4.1 役割

イベントゲージは、プレイヤー全員の開封行動によって蓄積される共有リソースである。
このゲージが満タンになっている場合、次ラウンド開始時に数字操作イベントを自動発火させる。

** 4.2 保持単位

- 保持単位は =GameSession= 単位
- プレイヤーごとには持たない

** 4.3 増加条件

- 各プレイヤーがマスを 1 つ開けたときに増加する
- 1 回の通常マス開封につき =+10= とする
- スキップ時は増加しない
- FREE マスの初期開封は対象外とする

** 4.4 発火条件

- イベントゲージが最大値以上になっていること
- 発火タイミングは「次ラウンド開始直後」

イベントゲージの最大値は固定値ではなく、次で計算する。

- =eventGaugeMax = 30 * 参加人数=

ここでいう参加人数は、=セッション開始時点のプレイヤー人数= である。

この値はセッション開始時に 1 回だけ確定し、進行中は再計算しない。

** 4.5 発火後の処理

- 数字操作イベントを 1 回だけ実行する
- イベントゲージは 0 にリセットする
- 余剰値は繰り越さない

** 4.6 今回の実装範囲で固定すること

- イベントゲージの最大値を持つ
- 現在値を保持する
- 開封時に増加する
- 満タン時に次ラウンド開始で自動発火する
- 発火後に 0 へ戻す
- 通常マス開封ごとの増加量は =+10= とする
- 最大値は =30 * 参加人数= とする

** 4.7 今回まだ固定しないこと

- バランス調整値

* 5. イベント発火モーダル仕様

** 5.1 役割

イベント発火モーダルは、プレイヤーがイベントを選択する UI ではない。
今回の段階では、サーバー側で自動的に発火したイベントの結果を全員へ見せるための表示 UI とする。

** 5.2 表示タイミング

- 次ラウンド開始時
- かつ、イベントゲージが満タンでイベントが発火した場合のみ

** 5.3 表示内容

最低限、次を表示する。

- イベントが発火したこと
- 元の抽選番号
- 変化後の確定番号
- 番号が変化していく演出

** 5.4 表示の性質

- プレイヤー入力は受け付けない
- =使う= / =使わない= のボタンは置かない
- 手札選択 UI は今回の段階では不要
- モーダル表示中は通常のカード操作を受け付けない

** 5.5 モーダル終了後

- モーダル表示が完了したら、確定番号を用いた通常のラウンド進行へ移る
- プレイヤーはその番号に対応するマスを開けるか、存在しなければスキップする

* 6. 数字操作機能の現段階での扱い

数字操作の API の細かい挙動は、今回の段階ではまだ固定しない。
ただし、システム全体を組み上げるために最低限の前提だけは固定する。

** 6.1 固定する前提

- サーバーが通常抽選した番号を =元の番号= とする
- イベント発火時、サーバーが =元の番号= を変化させる
- クライアントが開封判定に使うのは =変化後の確定番号= のみである

** 6.2 後で決めること

- どの API で数字操作を受け付けるか
- 足し引きの入力形式
- 範囲外や重複時の扱い
- 演出用レスポンスの詳細

* 7. 抽選番号管理の新データ構造

既存の =drawnNumbers= だけでは、
=元の抽選番号=
と
=イベント後の確定番号=
を区別できないため、データ構造を変更する。

今回の仕様では、次の概念を導入する。

** 7.1 rangeNumbers

- 抽選されうる番号の全体集合を表す配列
- デフォルトでは =1..75=

例:

#+begin_src ts
rangeNumbers = [1, 2, 3, ..., 75]
#+end_src

** 7.2 decidedNumbers

- そのラウンドまでに「最終的に確定した番号」を保持する配列
- クライアントが開封判定に使ってよい番号の履歴はこれである
- イベント発火がなかった場合は通常抽選番号が入る
- イベント発火があった場合は変更後の確定番号が入る

重要:

- 元の抽選番号ではなく、確定後の番号だけを格納する

** 7.3 drawableNumbers

- =rangeNumbers= から =decidedNumbers= の要素を除いた配列
- 次回抽選時の候補集合になる

定義イメージ:

#+begin_src ts
drawableNumbers = rangeNumbers.filter(
  (value) => !decidedNumbers.includes(value)
)
#+end_src

補足:

- 以前の案にあった =drawFromTheseNumbers= と同義だが、仕様上は =drawableNumbers= と呼ぶ

** 7.4 originalDrawnNumber

- そのラウンドでサーバーが最初に通常抽選した番号
- イベント発火時の演出や履歴に使う
- クライアントの開封判定には直接使わない

** 7.5 effectiveDrawnNumber

- そのラウンドで最終的に確定した番号
- クライアントがマス開封判定に使う番号
- =decidedNumbers= に追加されるのはこの値である

* 8. ラウンド開始時の抽選確定処理

次ラウンド開始時の番号確定処理は、次の流れとする。

1. サーバーは =drawableNumbers= から 1 つ数字を抽選する
2. 抽選した値を =originalDrawnNumber= とする
3. イベントゲージが満タンでなければ、=effectiveDrawnNumber = originalDrawnNumber= とする
4. イベントゲージが満タンなら、サーバー側で数字操作イベントを発火し、=effectiveDrawnNumber= を決定する
5. =effectiveDrawnNumber= を =decidedNumbers= に追加する
6. クライアントへ =effectiveDrawnNumber= を渡す
7. もしイベントが発火したなら、あわせて =originalDrawnNumber= も UI 演出用に渡す

* 9. クライアント側の挙動

** 9.1 イベントが発火しなかった場合

- 通常どおり抽選番号を表示する
- その番号をもとにマスを開けるか、スキップする

** 9.2 イベントが発火した場合

- モーダルを表示する
- =originalDrawnNumber= から =effectiveDrawnNumber= へ変わるアニメーションを描画する
- アニメーション完了後、=effectiveDrawnNumber= を使って通常の開封判定へ移る

** 9.3 開封判定

- カード上に =effectiveDrawnNumber= が存在すれば、そのマスを開けられる
- 存在しなければスキップ対象となる
- ここ以降は既存のゲーム進行へ合流する

* 10. サーバー状態の拡張方針

今回の実装で最低限必要になる追加状態は次のとおりである。

** 10.1 セッション単位で追加する状態

- =rangeNumbers=
- =decidedNumbers=
- =originalDrawnNumber=
- =effectiveDrawnNumber=
- =eventGauge=
- =eventGaugeMax=
- =eventTriggeredThisRound=

補足:

- =eventGauge= は現在値を保持する
- =eventGaugeMax= は固定値ではなく =30 * 参加人数= で決まる
- 通常マス開封時は =eventGauge += 10= とする

状態イメージ:

#+begin_src ts
type GameSession = {
  id: string;
  status: "waiting" | "in_progress" | "finished";
  phase: RoundPhase;
  round: number;
  rangeNumbers: number[];
  decidedNumbers: number[];
  originalDrawnNumber: number | null;
  effectiveDrawnNumber: number | null;
  eventGauge: number;
  eventGaugeMax: number;
  eventTriggeredThisRound: boolean;
};
#+end_src

** 10.2 プレイヤー単位で維持する既存状態

- =card=
- =isReadyForStart=
- =hasActedThisRound=

今回の実装範囲では、イベントゲージはプレイヤー個別には持たない。

* 11. 既存の drawnNumbers からの変更方針

既存の =drawnNumbers= は、これまで「抽選済み番号一覧」として使われていた。
しかし今後は「一度抽選されたが、イベントにより無効化された番号」が発生しうるため、このままでは不十分である。

そのため、今後の主たる仕様は次のように変更する。

- 既存の =drawnNumbers= 中心設計をやめる
- 最終確定済み番号の管理は =decidedNumbers= を正とする
- 次に引ける番号集合は =drawableNumbers= によって導出する
- =originalDrawnNumber= は補助情報として持つ
- クライアントへ主に渡すべき番号は =effectiveDrawnNumber= である

補足:

- 実装上、既存コードとの互換のために一時的に =drawnNumbers= を残す可能性はある
- ただし仕様上の正本は今後 =decidedNumbers= とする
